Prototype Architecture · v0.1

Render — the browser whose brain is an agent

A Chromium browser (not a web app) with a built-in agent, built-in opencli, and a built-in execution sandbox. You talk to a floating input box; the agent acts through three hands and answers in a structured, safe, interactive surface.

Render = llm/agent (brain) + opencli adapters (app hand) + sandbox (sandbox hand) + chromium CDP (human hand)

1 · The four organs

Brain — codex agent

codex app-server (codex-cli 0.136.0) driven over JSON-RPC-lite/stdio. Plans, reasons, decides which hand to move. Runs inside the sandbox. Validated in codex-opencli-poc/stage1.

App hand — opencli

1200+ adapters. API/public adapters run headless in the sandbox; browser adapters are routed to the human hand. The agent reaches sites & apps as deterministic CLI.

Sandbox hand — e2b / local

Isolated runtime for the brain + shell + API adapters + opencli-uncovered local execution. One SandboxProvider seam: local-seatbelt today, e2b for cross-end.

Human hand — chromium CDP

The in-process Chromium tabs the user actually sees & is logged into. Driven via webContents.debugger (CDP 1.3). Holds Plane-2 web auth. This is why Render is a browser.

3 · The user journey

1. User types into the floating input box (bottom-center, always present). 2. Main process opens a codex turn inside the sandbox; the agent panel (right) streams the event flow. 3. Agent moves a hand: · API/public need → opencli runs in the sandbox, returns structured data · logged-in site → opencli runs over CDP on the real browser tab · local execution → shell/code in the sandbox 4. Every agent reply renders through json-render, by message type: a. ux render — prose + structured results (non-blocking) b. ux form — submit to app layer (options · table · percent · inputs) c. ux confirm — reasoning block; route a choice to the human (Ask-User / approval) d. login — external auth; opencli login takes over a real Chromium tab

4 · Two auth planes (the trust model)

PlaneWhatWhere it livesReaches the sandbox?
Plane 1 — modelcodex / LLM provider keyMain process (harness), or e2b secret storeInjected to brain only; never to web tools
Plane 2 — web/appPer-site login (cookies, 2FA)Local Chromium profile onlyNever. Borrowed via CDP, never copied

The sandbox is powerful but credential-blind. Anything that needs the user's identity on a site is executed on the user's own browser through the human hand — so passwords and session cookies never leave the device. This is the PoC's proven hard line (browser adapters fail headless, exit 69) turned into a security guarantee.

8 · Decisions & rejected paths

Why Electron, not Tauri / WebView2 / native

Render must be a real browser (render arbitrary pages) and drive those exact tabs via CDP. Only Electron gives a uniform embedded Chromium with in-process CDP (webContents.debugger) across desktop OSes. Tauri/WebView2 use the OS webview — no uniform CDP surface. iOS/Android are future thin-clients that share the same protocol contracts and reach a remote sandbox; out of scope for this prototype.

Why the agent runs in the sandbox (not the main process)

Keeps the brain credential-blind and disposable, and makes the cross-end story (thin client + e2b brain) a config flip rather than a rewrite. The only thing pinned to the device is the human-hand CDP plane that holds web auth.

Why lift the opencli-ux contract but drop its localhost server

The render/form/confirm JSON shapes + the @json-render/shadcn catalog whitelist are the safe, proven surface and we keep them verbatim. But Render is the app, so the isolated-origin localhost server (which existed to give opencli-as-CLI a UI) is replaced by a first-class native agent panel over IPC.

Open seams tracked for later

e2b egress policy; CDP action-scoping (task_id + allowed_origins + TTL) to avoid unscoped browser root; profile isolation per account; tunnel auth for remote client-pull. Flagged in codex-opencli-poc/codex-review.md.
