Running OpenCLI's bridge capability inside Render (Electron 33)

Engineering feasibility — cheapest path to drive Render's own tabs with OpenCLI, login & cookies staying in Render. Skeptical eval of 4 routes + a 5th that the research surfaced.

TL;DR — Do not port the extension. OpenCLI already ships a supported "bring-your-own-browser" path: set OPENCLI_CDP_ENDPOINT=http://127.0.0.1:<port> and the CLI/daemon connects directly over CDP, bypassing both the Chrome extension and the daemon's /ext WebSocket. Render just needs to expose a CDP endpoint over loopback. That collapses the whole problem from "shim/fork/reimplement an MV3 service worker" to "launch Render's Chromium with a debugging port and point an env var at it."

The decisive finding (verify-this-first)

OpenCLI documents two env vars for exactly this use case:

VarDocumented meaning
OPENCLI_CDP_ENDPOINT"Chrome DevTools Protocol endpoint for remote browser or Electron apps" — e.g. http://127.0.0.1:9222
OPENCLI_CDP_TARGETFilter CDP targets by URL substring (e.g. detail.1688.com) — picks which tab/target to attach

OpenCLI's docs explicitly treat Electron apps as first-class CDP targets ("Desktop app adapters (Electron, via CDP): Cursor / Codex / ChatGPT App / Discord / …"). The pattern is uniform: launch the Electron app with --remote-debugging-port=<port>, verify with curl http://127.0.0.1:<port>/json/version, then export OPENCLI_CDP_ENDPOINT="http://127.0.0.1:<port>". Render is itself an Electron app, so it is one of these targets.

Sources: opencli README env table, docs/advanced/electron.md, docs/guide/electron-app-cli.md. When OPENCLI_CDP_ENDPOINT is set the CLI connects to the remote-debugging HTTP port directly "rather than using browser extensions or other intermediaries."

What the extension actually is (decoded from local install)

From /Users/drej/.opencli/extension-v1.0.19 (manifest + minified dist/background.js, cross-checked against extension/src/protocol.ts on GitHub):

  • MV3, background.service_worker = dist/background.js, type: module. Perms: debugger, tabs, cookies, activeTab, alarms, storage, tabGroups, downloads; hosts <all_urls>.
  • Connects out to ws://localhost:19825/ext (pings http://localhost:19825/ping; reconnect backoff 2–5s). All hardcoded in the SW global.
  • Core work = chrome.debugger.sendCommand (24 call sites), attach/detach/getTargets/onEvent/onDetach on leased tabs, plus chrome.tabs.*, chrome.cookies.getAll, chrome.downloads.*, chrome.tabGroups.*.
  • Protocol is a request/response envelope: Command{id, action, page?, code?, session?, surface, url?, op?...} → Result{id, ok, data?, error?, errorCode?...}. action ∈ {exec, navigate, tabs, cookies, screenshot, close-window, sessions, set-file-input, insert-text, bind, network-capture-start, network-capture-read, wait-download, cdp, frames}.

The extension is just a CDP shuttle: daemon ↔(WS /ext)↔ extension ↔(chrome.debugger)↔ Chrome tabs. The OPENCLI_CDP_ENDPOINT path deletes the two middle hops and lets the daemon speak CDP straight to a target. So everything the extension does, the daemon can do itself given a CDP endpoint — which is why route 5 below is essentially free.

Route-by-route evaluation

Route 1 — Load unmodified extension + shim chrome.debugger NOT VIABLE

Verdict: definitively no on Electron 33. Electron's extension layer is a partial, hardcoded reimplementation of the chrome.* surface in C++/JS — it is not Chrome. You cannot inject a JS chrome.debugger into a loaded extension's MV3 service-worker global:

  • There is no "extension preload" hook for the background service worker. session.preload/webContents preloads apply to pages/renderers, not the extension SW's isolated global.
  • Electron only injects the chrome.* APIs it has implemented; chrome.debugger isn't among them (0 occurrences in electron.d.ts, confirmed). Monkeypatching from outside the SW realm isn't reachable.
  • Even Electron's own MV3 SW support is shaky in this era — known gaps/bugs (chrome.scripting not implemented; SW APIs only gradually filled in, e.g. chrome.i18n fixed in 33.3.1). Standing up a reliable third-party MV3 background SW is itself a risk, before you even reach the missing debugger API.

Sources: Electron extensions docs, #37876 chrome.scripting not implemented in MV3, #34178 MV3 SW supported APIs don't function.

Route 2 — Fork the extension; replace chrome.debugger.* with an RPC to Render main WORKS, BUT WASTEFUL

Mechanically possible: source is open (extension/src/{background,cdp,protocol,identity}.ts, Vite build). You'd swap cdp.ts's chrome.debugger.sendCommand for a call that reaches Render main, which executes via webContents.debugger.sendCommand. Channel from an MV3 SW → main: a fetch()/WebSocket to a localhost server hosted in main is the only robust option (the SW has no Node and no IPC; content-script+preload bouncing is fragile and can't run on the SW's own behalf).

Why not: you still inherit Electron 33's flaky MV3 SW lifecycle, still load a third-party extension, still maintain a fork against upstream — to rebuild a CDP pipe that OPENCLI_CDP_ENDPOINT already gives you for free. Effort M–L, ongoing maintenance, no upside over route 5. Keeps everything in Render: yes. Recommend: no.

Route 3 — Reimplement the daemon /ext backend natively in Render main VIABLE, UNNECESSARY

The /ext protocol is highly decodable — the TS types are public in protocol.ts (Command/Result/Action above), and DAEMON_WS_URL etc. are literally the first lines of the shipped background.js. So you could have Render main open ws://localhost:19825/ext, send hello, accept leases, and service cdp/navigate/cookies/screenshot via webContents.debugger.

But: /ext is a private, versioned daemon↔extension contract (no public spec; "No WebSocket protocol documentation" in the docs) — it will drift between releases and you'd be chasing it. The OPENCLI_CDP_ENDPOINT path is the supported public seam that exists precisely so you don't reverse /ext. Pick route 5 instead. Effort M; risk: silent protocol drift on every opencli bump.

Route 4 — electron-chrome-extensions library DOESN'T HELP

It broadens Electron's chrome.* coverage (tabs, windows, contextMenus, etc.) but is DevTools/UX-focused and does not implement chrome.debugger for extensions. You'd be back to contributing a full chrome.debugger implementation upstream (large C++/JS surface, security-sensitive) — far more work than route 5, and still in service of route 1/2 which we've rejected. Not a shortcut.

Source: electron-chrome-extensions (npm), Electron extensions docs.

Route 5 (surfaced by research) — BYO CDP endpoint: OPENCLI_CDP_ENDPOINT → Render's Chromium RECOMMENDED

Render is Electron/Chromium, so it can expose the very CDP surface opencli already knows how to drive. Two sub-variants:

VariantHowTrade-off
5a — remote-debugging-portLaunch Render's Chromium with --remote-debugging-port=<port> bound to loopback; point OPENCLI_CDP_ENDPOINT at http://127.0.0.1:<port>. opencli hits /json/version, opens the target's WS, drives it.Lowest effort S. Standard Chromium /json discovery — exactly what opencli expects. Caveat: the port is an open CDP surface on loopback (any local process can attach) → must bind 127.0.0.1, ideally ephemeral/random port, and treat it as a local-trust boundary. Targets are renderer tabs only; main process isn't exposed unless you also pass --inspect (don't).
5b — webContents.debugger + tiny CDP proxyNo global debugging port. Render main owns each tab via webContents.debugger.attach('1.3') and exposes a minimal /json/version+per-target WS proxy on loopback that opencli connects to, forwarding frames to webContents.debugger.sendCommand.Effort M. Keeps the in-process model (no app-wide CDP port, finer control over which tabs are leasable), but you reimplement the small /json discovery shim opencli needs. Choose only if 5a's open-port posture is unacceptable.

Login/cookies stay in Render in both variants — opencli only attaches CDP to Render's existing, logged-in renderer sessions; it never gets its own browser profile. Cookie scraping happens via CDP (Network.getCookies/Storage) against Render's session, exactly as intended.

Spike plan for the top pick (5a)

  1. Kill-or-confirm the single biggest risk first (below) — does opencli actually attach and drive a target it discovers through OPENCLI_CDP_ENDPOINT pointed at an Electron/Chromium debugging port, with login state intact? Validate against Render before building anything.
  2. Launch a throwaway Electron 33 build (or Render itself) with --remote-debugging-port=0 bound to loopback; read the chosen port; curl http://127.0.0.1:<port>/json/version and /json/list to confirm targets enumerate.
  3. export OPENCLI_CDP_ENDPOINT="http://127.0.0.1:<port>" (+ OPENCLI_CDP_TARGET=<url-substring> to pin the tab). Run a trivial opencli browser command (navigate + screenshot) and confirm it acts on the Render tab.
  4. Verify a logged-in flow: sign in inside Render, then run an opencli command that reads cookies / hits an authed page — confirm session is reused and nothing leaks to a separate profile.
  5. Lock down the surface: bind 127.0.0.1 only, ephemeral port, don't pass --inspect, gate enabling the port behind an explicit Render setting; document it as a local-trust boundary. Strip the flag from any shipped/production build path.
  6. Wire port discovery into Render's launch and export OPENCLI_CDP_ENDPOINT for the opencli process Render spawns; add a health check (/json/version reachable) before issuing commands.

Biggest risk to kill first

Target discovery & attach semantics across the CDP-endpoint seam. The whole plan rests on opencli's OPENCLI_CDP_ENDPOINT path doing real /json target discovery + per-target WS attach against an arbitrary Chromium debugging port — not assuming a flow that only the extension provided (tab leasing, tabGroups, downloads, bind). If opencli's CDP-endpoint mode supports only a subset of actions (e.g. exec/navigate/screenshot/cookies but not bind/downloads/network-capture), some adapters may degrade. Confirm before building: point OPENCLI_CDP_ENDPOINT at a plain Chromium started with --remote-debugging-port and exercise the exact opencli commands Render needs end-to-end. If it works there, Render (same Chromium) will work; if an action is unsupported in CDP-endpoint mode, that's your gap to close (and only then does route 3/5b come back into scope).

Open items worth a follow-up read

  • Read extension/src/cdp.ts + background.ts on GitHub to enumerate which actions depend on extension-only APIs (tabGroups, downloads) vs pure CDP — tells you exactly what, if anything, the CDP-endpoint mode can't do.
  • Confirm in opencli source which actions the OPENCLI_CDP_ENDPOINT code path implements (vs the /ext path) — this is the precise scope of route-5a coverage.
