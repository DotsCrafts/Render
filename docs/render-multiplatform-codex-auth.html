Render — 多端部署 + Codex 认证架构

mac / windows / iOS / android 部署优先级 · Codex OAuth/API-key 登录 + 可替换 base-url（含移动端）· 基于当前 Electron + agent-bridge + e2b seam 现状
一句话结论

Render 的难点不是"再写一个客户端"，而是 brain（codex）和 opencli 都是 Node 进程，移动端跑不了。所以架构必须沿三条平面拆开： ① 大脑（codex 推理） · ② 浏览面 + 凭证（Plane-2 登录态） · ③ 协议层。桌面端三者本地同机；移动端把 ①③ 推到远端、② 留在设备上的原生 WebView。 幸运的是 sandbox（local-seatbelt / e2b）和 OPENCLI_CDP_ENDPOINT relay 这两个 seam 已经存在，移动端是把它们接到远端，而不是重写。

1 · 三平面模型（这是一切的地基）

① Brain

codex app-server（Node）。做推理、调 opencli、产出 json-render UX。

桌面：本地 sandbox（local-seatbelt）。
移动：远端（e2b / Render cloud），手机不跑 Node。

② Browser + 凭证

浏览面 + 登录态（Plane-2）。必须留在用户设备上，密码/cookie 永不进 sandbox。

桌面：Electron WebContentsView + persist:render。
移动：原生 WKWebView / Android WebView。

③ 协议层

@render/protocol + agent-bridge + json-render。传输无关。

桌面：进程内 IPC。
移动：WebSocket（手机 ↔ 远端 brain）。同一套帧。

2 · 拓扑：桌面 vs 移动

① brain 本地 ② 浏览/凭证 本地 ③ 协议 进程内

现状（已实现）。Windows/Linux 基本是同一份 Electron 代码，主要是打包/签名 + 路径差异。

① brain 远端 ② 浏览/凭证 留手机 ③ 协议 over WS
移动端最硬的两块：
  • opencli bridge：手机不能跑 daemon。远端 opencli 想驱动"手机里的 WebView"需要把 bridge 的 /ext 协议反向接到手机 WebView 的 CDP-like 桥（WKWebView 用 WKUserContentController/JS 注入，Android 用 WebViewCompat + DevTools 协议）。这是移动端工作量的大头。
  • Plane-2 登录态：必须在手机本地 WebView 完成（同桌面的"在 Render 自己标签里登录"），cookie 留 Keystore/WebView 数据区，永不发往远端 brain。

3 · 部署优先级

优先级平台路径主要工作
P0macOS现有 Electron固化当前架构（tab-group/登录/profile-bind 已完成）；签名 + 公证 + 自动更新。
P0Windows同份 Electron≈免费拿下。打包(NSIS/MSIX)+签名；验证 CDP 端口、keychain→safeStorage(DPAPI)、opencli daemon 路径。
P1核心解耦重构（非平台）把"①brain ②浏览 ③协议"三 seam 抽成传输无关接口（local IPC ↔ WS 二选一）。这是移动端的前置条件，桌面端也受益。
P1Linux同份 ElectronAppImage/deb；近免费，顺带产物。
P2iOS原生薄客户端 + 远端 brainWKWebView 浏览面 + json-render 渲染器 + WS client；远端 codex；WebView↔bridge relay；OAuth via ASWebAuthenticationSession。
P2Android原生薄客户端 + 远端 brain同 iOS，WebView + Custom Tabs/App Links 做 OAuth；Keystore 存凭证。
P3云端 brain 托管Render cloud给"不想自带 e2b key"的移动用户提供托管 brain + 凭证保险库。

原则：桌面先吃饱（P0），再把解耦做实（P1），移动端才不是重写而是接 seam（P2）。 移动端不要在解耦完成前硬上。

移动端技术栈选型

方案优劣评估
原生 Swift + KotlinWebView/Keychain/OAuth/后台 最可控；Plane-2 安全性最高两套代码推荐。浏览面+凭证是核心且平台差异大，值得原生。UI 壳薄（大多是 json-render 渲染器）。
React Native共享渲染器/协议 client；JS 与现有栈同源WebView 深度驱动(CDP桥)需原生模块；OAuth 仍要原生次选。若想最大化复用 json-render(TS) 渲染器可考虑。
Capacitor / Flutter快WebView 作为"被 agent 驱动的浏览面"控制力不足；relay 桥更别扭不建议——和"Render 是浏览器"内核冲突。

4 · Codex 认证：OAuth / API-key / base-url

现状：codex 读 ~/.codex/auth.json（你的是 OPENAI_API_KEY）+ config.toml（你已用自定义 base_url=api.ai.ifunk.cn, wire_api=responses）。Render 现在只是把它们拷进临时 CODEX_HOME。需求 = 把这套变成 Render 里一等的登录/设置，加 OAuth，并跨到移动端。

三种认证模式（统一收敛到 codex 的 auth.json + config.toml）

模式用户输入落到哪说明
API Key粘贴 keyauth.json: {OPENAI_API_KEY}最简单。你当前就是这个。
OAuth浏览器授权登录auth.json: token + refreshChatGPT 式 OAuth（codex 已支持 chatgpt token 形态）；需做 PKCE + 刷新。
Base-URL 替换endpoint + wire_api + 鉴权方式config.toml [model_providers.X]自托管/代理/三方兼容端点。与上面两种正交（任意 provider 都可配 key 或 oauth）。

设置模型（Render 一等公民）

新增一个 Provider 设置面（json-render 表单即可，复用现有 UX surface）：

  • Provider 名Base URLwire_api: chat | responses认证: api-key | oauth
  • 提交 → Render 主进程把它写进会话级 CODEX_HOME 的 config.toml + auth.json（沿用现有 prepareCodexHome，从"拷用户的"升级为"按 Render 配置生成"）。
  • 多 provider 预设 + 一键切换（你在 ifunk / 官方 / 自托管间切）。

凭证存哪（按平台）—— 安全是红线

平台静态存储注入 brain 的方式
macOSKeychain / Electron safeStorage会话启动时解密 → 写入临时 CODEX_HOME（本地 sandbox）→ 用完清理
WindowssafeStorage(DPAPI)同上
iOSKeychain会话握手时经 TLS 发给远端 brain 的临时 CODEX_HOME，不落服务端磁盘、随沙箱销毁
AndroidKeystore / EncryptedSharedPreferences同 iOS
移动端凭证的两种托管选择（需你拍板）：
  • BYO-key（自带）：key/token 只存手机 Keychain，每次会话经 TLS 临时发给远端 brain，服务端不持久化。最简单、最安全、零信任服务端。← 推荐起步。
  • Render-brokered：Render 后端做 OAuth + token 保险库，手机只认证到 Render。体验顺滑但 Render 成了凭证托管方（合规/信任成本）。留给 P3。

OAuth 在移动端怎么跑

iOS

ASWebAuthenticationSession + PKCE，render://oauth-callback 深链回跳，token 存 Keychain。

Android

Chrome Custom Tabs + App Links + PKCE，深链回 App，token 存 Keystore。

桌面端 OAuth 走系统浏览器或 Render 自己的标签（persist:render），回跳 localhost loopback。注意：codex 自带的 preferred_auth_method 与自定义 base_url 同用时，OAuth 端点需 provider 支持——第三方兼容端点常常只支持 api-key，OAuth 主要对官方/支持 OIDC 的 provider 有意义。
