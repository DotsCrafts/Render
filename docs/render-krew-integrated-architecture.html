Render × krew-dev — 整体架构

复用 krew-dev 已部署在 testnet3 的 e2b + krewhub + krewauth + agent-vault;Render 把 codex 大脑搬进 e2b sandbox,桌面/移动统一

核心洞察

  • Render ≈ 又一个 krew 客户端 —— 远端大脑这条路的基础设施(e2b 集群、编排 krewhub、身份 krewauth、凭证出口 agent-vault)krew-dev 已经在 testnet3 部好了,不用重建。
  • 唯一根本差异:大脑跑哪。 krew 现在用 krewcli 在本机跑 brain;Render 要把 codex app-server 跑进 e2b sandbox(移动端必须如此)。
  • 传输大半已就绪: Render 的 E2bSandbox 已经能用 e2b SDK 把 codex 的 stdio 隧道出来 —— 桌面端设好 E2B_API_KEY+E2B_DOMAIN 指向这个集群,大脑立刻能在 e2b 远端跑。缺的是:① codex 的 e2b 模板 ② 移动端用的公网 WS 网关 ③ Plane-2(浏览器)回连客户端的 relay。

2 · 三种运行模式

桌面本地(今天就是):codex 在用户机器 local-seatbelt 跑,Plane-2 在 Electron 标签。零基础设施成本,不碰 krew。模型走 base_url(你的 ifunk)。→ 这是 Free 层 / 获客漏斗。

桌面远端:Render 设 E2B_API_KEY+E2B_DOMAIN 指向 krew 的 e2b 集群,E2bSandbox 在 e2b 里 spawn codex,stdio 经 e2b SDK 隧道回桌面。Plane-2 仍在本地 Electron。前提:e2b 暴露给笔记本(ALB/VPN)。 用于重负载 / 验证 Plane-① 远端化。

移动:手机起不了 Node / 够不到 VPC。薄客户端 → krewhub 网关(公网 hub.cookrew.dev,krewauth 鉴权)→ 在 e2b 里起 codex sandbox → WS 隧道把 codex app-server 协议中继回手机渲染。Plane-2 留手机原生 WebView,远端 opencli 经 relay 驱动它。模型 key 经 agent-vault 注入。

3 · 凭证模型(双类 —— Plane-2 永不出设备)

关键:凭证分两类,隔离边界完全不同。e2b / 远端大脑只短时接触 Plane-1,Plane-2 永远留在用户设备。

类别是什么存哪碰 e2b?
Plane-2 网站登录态dianping/github 的 cookie/session用户设备本地:桌面 persist:render / 手机 WebView cookie store永不
Plane-1 模型凭证codex 的 API key / OAuth token桌面 safeStorage;移动经 agent-vault 临时注入远端 sandbox短时·不落盘

"要不要给手机端单独设权限?" —— 不用为"网站登录态"在 e2b 里设权限,因为它根本不在 e2b。per-user 隔离已由三层现成机制保证:krewauth 身份(每人一 JWT)+ e2b sandbox 每会话独立 microVM + agent-vault per-vault token 作用域。无需另造"手机端权限系统"。

4 · 浏览器驱动后端对照(CDP vs 原生 WebView)

移动端拿不到干净的、App 内可用的 CDP:iOS WKWebView 是 WebKit,根本没有 CDP(只有 Safari/USB 的 Web Inspector,App 内不可程序化);Android WebView 内核是 Chromium,DevTools 仅在 adb chrome://inspect 调试通道暴露,不是生产自动化通道。但不需要 CDP —— bridge 的 /ext 协议是在 navigate/exec/cookies 这一层抽象,后端可按平台可插拔:桌面用 CDP,移动用原生 WebView API。

5 · 会话生命周期(移动 / 远端)

  1. 登录:客户端 → krewauth(device flow / passkey)→ JWT。
  2. 开会话:客户端 → krewhub POST /sessions(带 JWT)→ krewhub 用 e2b client 建一个 codex 模板的 sandbox。
  3. 注入凭证:krewhub 经 agent-vault 给 sandbox 配 HTTPS_PROXY + Render provider 设置(base_url/auth)写进 sandbox 的 CODEX_HOME。模型 key 永不进 brain 明文。
  4. 起大脑:sandbox 内跑 codex app-server;krewhub 把它的 stdio 经 WS 隧道桥到客户端的 CodexClient。
  5. 渲染:codex 产出 json-render UX 帧 → WS → 手机渲染器。HITL(approve/login)→ 帧回传。
  6. Plane-2:大脑要登录态站点时,opencli 经 relay 驱动手机本地 WebView(同桌面"在 Render 自己标签登录")—— cookie 留设备。
  7. 结束/挂起:idle → e2b pause(省算力);重连 → resume 同一 sandbox。

7 · 推进顺序

阶段交付
现在Phase A(进行中):桌面本地大脑 + Render-managed codex provider/auth(OAuth/api-key/base-url)。零基础设施。
下一步G4 + G1:暴露 e2b 给桌面(VPN/ALB)+ 建 codex 模板 → 桌面"远端大脑"模式跑通(复用现有 E2bSandbox stdio 隧道)。这是最小可验证的远端化,不碰移动。
再下G2 + G3:krewhub WS 会话隧道 + Plane-2 relay → 远端大脑能驱动客户端浏览器。
最后G5:移动外壳(iOS/Android)接上 G2/G3 → 移动端端到端。
关键判断: 因为 krew 这套已经部署,Render 的远端化不是"从零搭后端",而是"把 codex 大脑塞进已有的 e2b + 复用 krewhub/krewauth/agent-vault"。最小验证路径 = G4+G1(桌面远端大脑),它不依赖移动外壳,且复用 Render 已有的 E2bSandbox stdio 隧道 —— 风险最低、最快证明整条链路。模型成本仍走 BYO-key(见 e2b 成本测算)。
