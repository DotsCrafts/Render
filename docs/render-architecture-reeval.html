Render · Architecture Decision Record

Should Render integrate a Chrome plugin system — and is the whole architecture sound?

Re-think prompted by one hard invariant: Render is a browser. The agent must never jump to another browser. Login & cookie-scraping happen inside Render — or building a browser is pointless.

Verdict (TL;DR)

  • Don't build a plugin system to run opencli's extension. Electron structurally cannot host it — the extension lives on chrome.debugger, which Electron's extension API does not implement, cannot shim into an MV3 service worker, and has no roadmap for. A full plugin system = a Chromium fork = a permanent ~monthly rebase tax + a browser-platform team. Overkill for our actual need.
  • "Just point OPENCLI_CDP_ENDPOINT at Render" does NOT solve it. opencli's own source routes only registered Electron-app adapters through CDP; every cookie/site adapter (dianping, taobao, zhihu…) is hardwired to the daemon+extension bridge. Proven below.
  • The right move is the inverse of a plugin system: make Render the browser opencli drives. Render serves opencli's daemon browser-backend (/ext) over its own tabs via webContents.debugger (CDP). No extension, no fork, no Chromium fork. The leak to system Chrome becomes structurally impossible.
  • The brain / hands / sandbox / json-render spine is sound and worth keeping. The two-CDP-plane split, the login-detector, the apps.yaml dance and the profile pin are all symptoms of one seam — cookie adapters binding to a backend Electron can't host. Fix the seam; the complexity collapses.

1 · How real browsers run Chrome extensions (and what that costs)

The capability Render needs is the hardest one: an MV3 extension that drives tabs via chrome.debugger/CDP. That is exactly the API the shim ecosystem cannot provide and Electron has no plan for. A Feb 2026 Electron bug (#49830) shows the structurally-identical Claude in Chrome extension failing to load in Electron for precisely this reason.

Telling datapoint: Wavebox — a productivity browser whose product is extensions — abandoned Electron for a native Chromium foundation specifically because "Electron lacked essential browser-type functionality such as PDF rendering and extension support." SigmaOS (hand-rolled its own extension API on WebKit) is the cautionary tale: real compatibility gaps land exactly on the advanced APIs (debugger, sidePanel, tabGroups) we'd depend on. Building your own plugin system reproduces those gaps; inheriting Chromium's avoids them — at fork cost.

2 · The decisive fact: opencli only CDP-drives registered apps, never cookie sites

This settles the cheap-path debate. From opencli's installed source (@jackwener/opencli/dist/src/runtime.js):

export function getBrowserFactory(site) {
  if (site && isElectronApp(site)) return CDPBridge;   // honors OPENCLI_CDP_ENDPOINT
  return BrowserBridge;                                  // daemon + MV3 extension (port 19825)
}
  • OPENCLI_CDP_ENDPOINT is consumed only by CDPBridge (browser/cdp.js), and CDPBridge is chosen only when isElectronApp(site) — the registry of Electron apps (cursor, codex, trae, … and our hand-registered render).
  • Every cookie / site adapter (dianping, taobao, zhihu, weibo, xiaohongshu, …) falls to BrowserBridge → the daemon → the extension in some Chrome. There is no per-site override to force CDP.
  • This matches our live evidence: with OPENCLI_CDP_ENDPOINT set in the sandbox, opencli dianping search still returned AUTH_REQUIRED … sign in to dianping.com in this profile — i.e. it used the bridge profile, not Render's tab.

Consequence: the only way a cookie adapter can drive Render is if Render is the browser the daemon's bridge talks to. Pointing an env var at Render, hosting the extension in Render, and forking the extension are all either impossible (§1) or don't apply (§2). That leaves one shape.

3 · Re-evaluating Render's architecture

Sound — keep

  • Codex brain over JSONL app-server: clean, swappable process boundary.
  • SandboxProvider seam (local-seatbelt ↔ e2b): genuinely provider-agnostic.
  • Real CDP on our own tabs via webContents.debugger — the asset everything should build on.
  • json-render as the only renderer contract: catalog-whitelisted, structure-as-injection-boundary.
  • Trust boundary: Plane-1 model keys in main; Plane-2 site creds never enter the sandbox.

Shaky — the single leak

  • Assumption "opencli will drive Render's browser" only holds for the bespoke render get/text app adapter — not for cookie sites.
  • The relay + two CDP planes + apps.yaml registration + OPENCLI_PROFILE pin + the new login-detector exist only to route around opencli's cookie backend living outside Render.
  • It's architectural, not patchable: no env var / yaml / prompt makes a third-party adapter abandon its hardwired bridge.

Recommendation — Option A: "be the browser backend"

Render natively serves the contract opencli's cookie adapters speak to the daemon, and backs every CDP target with webContents.debugger on Render's own tabs (Render's profile, where logins live). Cookie adapters then drive Render — the same property our render get/text adapter already proves, generalized to all sites. It needs no extension, no fork, no Chromium fork, and reuses the relay + /json discovery already built. After it lands, the two CDP planes converge into one and the relay's only remaining job is the e2b tunnel — its actual purpose.

5 · What collapses once A lands

  • opencli-auth.ts (the AUTH_REQUIRED login-detector added this session) — a dead end under A: it scrapes the agent's narrated command stream to retrofit a login card that the router already produces, and logging into a browser that isn't Render never helped anyway. Delete it once A lands.
  • Two-plane CDP split / cdpEndpoint fallback in index.ts — two endpoints for what becomes one backend. Collapses.
  • apps.yaml registration + embedded render get/text — keep as the proof that the CDP route works; retire as a product mechanism (it mutates global ~/.opencli from app startup — a footgun). Fold its insight into A.
  • OPENCLI_PROFILE='default' pin — papers over opencli's profile model; unnecessary once Render owns the session.
  • Correction: the earlier "apps.yaml port 9477 vs 9333" alarm was wrong — prod is 9333 consistently; 9477 is only the test harness default in nav-guard.e2e.mjs. Align the test default; no prod bug.

6 · Decisive spike & top risks

Before committing to A, kill-or-confirm the one unknown with one cookie adapter:

  1. Stand up a minimal /ext backend in Render main: accept the daemon's WebSocket, answer hello, and execute cdp/navigate/cookies actions against a Render WebContentsView via webContents.debugger.sendCommand.
  2. Register Render as the daemon's active bridge profile; run opencli dianping search and confirm it drives a Render tab (and that login persists in Render's own partition).

Risk 1 — protocol drift

The /ext contract is opencli-private & minified. If it isn't cleanly separable from the extension, A's effort moves toward B. Mitigate: spike one adapter first; pin the opencli version.

Risk 2 — profile/session identity

Logins must persist in Render's own Chromium partition and be the session opencli drives. Today tabs use default partitions. Mitigate: design a persistent partition model alongside A.

Risk 3 — e2b tunnel

When the brain runs remotely, the cookie path must reach Render's backend over the relay tunnel — built but never exercised on this path. Mitigate: validate the remote case explicitly.
