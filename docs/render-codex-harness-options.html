云化 codex —— 4 方案跨端适配对比

把 codex 大脑放进 e2b、跨桌面/移动暴露给薄客户端,哪种 harness 协议最合适

先把两层分开(这是看懂对比的钥匙)

协议层(codex 说什么话)和传输层(怎么跨端送到客户端)是正交的两件事:

  • 方案 1/2/3 是「协议」之争 —— App Server vs ACP vs MCP,三者本质都是 stdio JSON-RPC。所以"跨端"对它们是同一个传输问题(都要把 stdio 隧道出去),真正的差异在交互保真度和提供方可换性。
  • 方案 4(opencli codex adaptor)其实是「传输/打包」想法 —— 复用我们已经把跨端解决掉的 opencli `/ext` 会合通道。它不替代协议,而是承载协议。
  • 关键事实:OpenAI 自己试过 codex-as-MCP 又放弃了 —— 因为"streaming diffs / 审批流 / 线程持久化映射不进 MCP 的工具模型"。这正是 Render 需要的东西。

2 · 协议 × 传输:它们其实是组合关系

3 · 结论 & 推荐

  • 主大脑协议 = ① App Server。 Render 已建在它上面,流式/审批/线程全保真(MCP 装不下、ACP 打折)。OpenAI 官方也推荐它做全保真嵌入。不换。
  • 跨端传输 = ④ 的好点子 + WS 网关。 把 App Server 的帧装进 opencli `/ext` 会合通道 —— 这样 Plane-1(大脑)和 Plane-2(浏览器)共用一条已解决跨端、NAT 友好、设备拨出的通道(统一鉴权/重连)。移动端再加 krewhub WS 网关。注意:别把 codex 做成"命令适配器"(那会丢交互流),会合只当传输,里面跑的还是 App Server 流。
  • ③ MCP 反向用:把 codex 暴露成 MCP 给 krew 的 agent 图组合(krewhub 的 AgentHand/delegate)—— codex 当子 agent,与"Render 主交互大脑"并存,不冲突。
  • ② ACP 当战略对冲: 等真要"一个 Render harness 跨 codex/claude/gemini 切换"时,再把 ACP 作为可选外层协议引入。现在不做。

一句话:协议选 App Server(保真),传输借 opencli 会合(复用已解决的跨端),MCP 留给 krew 做组合,ACP 留作以后换脑的对冲。

为什么 #4 当传输而不当协议: opencli 适配器是"跑命令拿结果",而 codex 会话是长连接、流式、带 HITL server-request。把大脑硬塞进命令模型 = 丢掉交互;但 opencli 的 `/ext` 会合通道(设备拨出 + daemon 路由)恰好是我们已经验证跨端、NAT 友好的传输 —— 让它多开一条"brain channel"承载 App Server 流即可,一举解决"脑和浏览器两条跨端链路"的统一。
